## 1. 项目概述

本项目旨在将精选的基督徒书报（神学注释、教义著作、讲道集等）构建为一个**封闭式 AI 检索知识库**。用户可以用自然语言提问，系统在指定书目范围内检索并引用原文作答。

**核心价值**：
- 互联网上神学信息良莠不齐、教派观点相悖，本系统**只从经过筛选的书目中回答**，确保来源可靠
- 每条回答附带原文引用，用户可核对出处
- 支持跨书目的概念追踪（如"同一个神学概念在不同作者之间的观点对比"）

---

## 2. 核心需求与约束

| 需求 | 说明 |
|------|------|
| **封闭知识源** | 只使用提供的书籍，不引用互联网或模型自身训练知识 |
| **强引用溯源** | 每条回答必须标注来源书目及段落，便于核对 |
| **大文本量** | 多本书籍，PDF 排版复杂，需高质量文档解析 |
| **跨书关联** | 支持"A 书与 B 书对同一主题的不同观点"类问题 |
| **双语支持** | 当前中文 + 英文，未来扩展西班牙语、韩语 |
| **拒绝幻觉** | 知识库中没有的内容，明确回答"无法从现有书目中找到" |

## 3. RAG 框架选型

### 3.1 候选框架对比

| 框架 | 文档解析 | 引用溯源 | 跨文档关联 | 上手难度 | 本地部署 |
|------|---------|---------|-----------|---------|---------|
| **RAGFlow** | ✓✓ DeepDoc 引擎 | ✓✓ 原文快照 | ✓ 基础支持 | 中 | ✓ |
| UltraRAG | ✓ | △ | ✓✓ Pipeline | 高（偏研究） | ✓ |
| Dify | △ | △ | △ | 低 | ✓ |
| AnythingLLM | △ | △ | △ | 最低 | ✓ |
| LangChain/LlamaIndex | ✓✓ | ✓✓ | ✓✓ | 高（需写代码） | ✓ |

### 3.2 为什么选 RAGFlow

**文档解析**：神学书籍 PDF 通常包含复杂排版（脚注、多栏、引用块）。RAGFlow 内置 DeepDoc 引擎，专门处理非结构化文档，能识别文档逻辑结构而非简单按位置切割文本。其他框架直接分块，容易把脚注和正文混在一起。

**强引用机制**：RAGFlow 是所有候选框架中引用溯源能力最强的，每条回答附带**原文快照截图 + 书目来源**，这对神学研究中"一字之差含义迥异"的场景至关重要。

**封闭知识源**：RAGFlow 可完全离线部署，彻底切断外部知识污染，完全符合"只从提供的书目回答"的要求。

**为什么不选 UltraRAG**：UltraRAG 的核心优势（模型微调、自动生成训练数据、复杂研究 Pipeline）在本项目中用不上。本项目不需要训练专属模型，只需高质量的检索与引用。

### 3.3 GraphRAG 的配合作用

RAGFlow 默认是**向量相似度检索**（找语义相近的文本块），适合"这本书里有没有说过 X"类问题。

但本项目还需要处理：
- "约翰·加尔文和司布真对恩典的理解有何不同？"
- "这个教义观点在哪几本书里被提及？"

这类**跨书目概念关联**问题需要 GraphRAG 补充：

```
向量检索（RAGFlow）    →  找到相关文本片段
     +
知识图谱（GraphRAG）   →  追踪概念在不同书籍中的关联
     ↓
综合回答：引用原文 + 展示不同作者的观点脉络
```

GraphRAG 会将书籍中的实体（人物、教义概念、圣经章节）和它们之间的关系构建成图，使检索能沿概念脉络而非单纯相似度展开。

## 4. LLM 模型选型

### 4.1 理解 RAG 的 Token 消耗模式

RAG 查询并非把整本书发给模型，实际 token 构成为：

```
每次查询消耗 ≈
  系统提示（~200 tokens）
  + 检索到的原文片段 × 5条（~3,000 tokens）
  + 用户问题（~50 tokens）
  + 模型回答（~500 tokens）
─────────────────────────────
总计约 3,750 tokens/次查询
```

### 4.2 API 调用方案对比

| 模型 | 输入价格 | 输出价格 | 中文能力 | 英文能力 | 未来多语言 | RAG 适配性 |
|------|---------|---------|---------|---------|-----------|-----------|
| **DeepSeek V3** | $0.27/M | $1.10/M | ✓✓ 优秀 | ✓✓ 优秀 | ✓✓ | ✓✓ 遵循指令强 |
| DeepSeek R1 | $0.55/M | $2.19/M | ✓✓ 优秀 | ✓✓ 优秀 | ✓✓ | ✓ 推理更深但较慢 |
| GPT-4o mini | $0.15/M | $0.60/M | ✓ 良好 | ✓✓ 优秀 | ✓✓ | ✓✓ |
| Gemini 1.5 Flash | $0.075/M | $0.30/M | ✓ 良好 | ✓✓ 优秀 | ✓✓ | ✓ |
| Claude Haiku 4.5 | $1.00/M | $5.00/M | ✓ 良好 | ✓✓ 优秀 | ✓✓ | ✓✓ |
| Claude Sonnet 4.6 | $3.00/M | $15.00/M | ✓✓ 优秀 | ✓✓ 优秀 | ✓✓ | ✓✓ 最强 |

> *价格单位：每百万 tokens（USD），数据截至 2026 年 3 月*

**API 首选：DeepSeek V3**
- 中英文综合能力最均衡（在中文 benchmark 上优于 GPT-4o mini）
- 价格为同级模型中最低
- 遵循"只用提供上下文回答"指令的能力强，幻觉率低
- 原生支持西班牙语、韩语，未来扩展无需换模型

**API 备选：GPT-4o mini**
- 若书目以英文为主，GPT-4o mini 英文理解略强
- 价格极低，月费与 DeepSeek V3 接近

**深度推理查询（可选）：DeepSeek R1**
- 用于"比较不同作者的神学立场"类复杂问题
- 成本约为 V3 的 2 倍，普通检索无需使用

### 4.3 本地部署方案对比

| 模型 | 所需显存 | 推荐显卡 | 参考价格（国内） | 推理速度 | 中英文能力 |
|------|---------|---------|----------------|---------|-----------|
| **Qwen3-14B**（4bit量化） | 10~12 GB | RTX 4060 Ti 16G | ≈ ¥3,000 | ~20 token/s | ✓✓ |
| Qwen3-32B（4bit量化） | 20~22 GB | RTX 4090 24G | ≈ ¥12,000 | ~12 token/s | ✓✓ 更强 |
| DeepSeek-R1-Distill-14B | 10~12 GB | RTX 4060 Ti 16G | ≈ ¥3,000 | ~18 token/s | ✓✓ |
| Llama 3.1 8B | 6~8 GB | RTX 3060 12G | ≈ ¥1,500 | ~30 token/s | ✓（英文为主）|

**本地首选：Qwen3-14B**
- 当前开源模型中中英文双语综合能力最强之一
- 14B 参数对 RAG 任务已完全够用（RAG 是检索+归纳，非推理密集型任务）
- RTX 4060 Ti 16G 即可流畅运行，性价比最高
- 支持"思考模式"开关：普通检索关闭以提速，复杂对比分析时开启
- 未来支持西班牙语、韩语，无需更换模型

**不推荐 Llama 系列**：中文能力弱于 Qwen3，不适合以中文书籍为主的知识库。

## 5. 推荐方案

### 方案 A：API 调用（推荐用于启动阶段）

```
书籍 PDF / TXT
    ↓
RAGFlow（文档解析 + 向量检索 + 引用）
    + GraphRAG（概念关联图谱）
    ↓
DeepSeek V3 API（推理生成）
    ↓
带原文引用的回答
```

**适合**：快速验证效果、无 GPU 硬件、预算有限

### 方案 B：完全本地部署（推荐用于长期运行）

```
书籍 PDF / TXT
    ↓
RAGFlow（本地部署）
    + GraphRAG（本地运行）
    ↓
Qwen3-14B（Ollama 本地部署）
    ↓
带原文引用的回答（数据不出本机）
```

**适合**：数据隐私要求高、长期高频使用、有 RTX 4060 Ti 以上硬件

### 方案 C：混合部署（推荐的最终形态）

- 日常检索：Qwen3-14B 本地运行
- 复杂对比分析：调用 DeepSeek R1 API 补充深度推理
- 数据：完全本地，不出境

## 6. 费用分析

### 6.1 API 方案月费估算

**使用假设**：个人使用，每天 50 次查询，每次约 3,750 tokens

```
每月查询次数：50 次/天 × 30 天 = 1,500 次
每月输入 tokens：1,500 × 3,200 = 4.80M
每月输出 tokens：1,500 × 500 = 0.75M
```

| 模型 | 输入费用 | 输出费用 | **月合计** |
|------|---------|---------|-----------|
| DeepSeek V3 | $1.30 | $0.83 | **≈ $2.1** |
| GPT-4o mini | $0.72 | $0.45 | **≈ $1.2** |
| Claude Haiku 4.5 | $4.80 | $3.75 | **≈ $8.6** |
| Claude Sonnet 4.6 | $14.40 | $11.25 | **≈ $25.7** |

> **结论**：DeepSeek V3 月费约 **$2**，即便每天查询量翻 3 倍，月费也不超过 $7。

| 阶段 | API 方案（DeepSeek） | 本地方案（Qwen3-14B）|

## 7. 实施流程

### Phase 0：环境准备（第 1 天）

```bash
# 安装 Docker（RAGFlow 依赖）
# macOS
brew install --cask docker

# 克隆 RAGFlow
git clone https://github.com/infiniflow/ragflow.git
cd ragflow/docker
docker compose up -d

# 访问 RAGFlow WebUI
open http://localhost:80
```

**检查项**：
- [ ] Docker 正常运行
- [ ] RAGFlow WebUI 可访问
- [ ] 注册账号并登录

---

### Phase 1：配置 LLM（第 1 天）

**使用 DeepSeek V3 API（推荐起步）**：

1. 访问 [platform.deepseek.com](https://platform.deepseek.com) 注册并获取 API Key
2. 在 RAGFlow 中进入 `Settings → Model Providers`
3. 选择 `DeepSeek`，填入 API Key
4. 设置默认模型：
   - Chat Model：`deepseek-chat`（V3）
   - Embedding Model：`BAAI/bge-m3`（RAGFlow 内置，支持中英文）

**若使用本地 Qwen3-14B**：

```bash
# 安装 Ollama
curl -fsSL https://ollama.ai/install.sh | sh

# 下载模型（约 8GB）
ollama pull qwen3:14b

# 启动服务（监听 11434 端口）
ollama serve
```

在 RAGFlow 中配置：
- Model Provider：`Ollama`
- Base URL：`http://localhost:11434`
- Chat Model：`qwen3:14b`

---

### Phase 2：构建知识库（第 2~7 天）

**2.1 创建知识库**

在 RAGFlow WebUI：
1. `Knowledge Base → Create`
2. 命名：`Christian Literature`
3. Embedding Model：`BAAI/bge-m3`（中英双语嵌入模型）
4. Chunk Method：选择 `Book`（针对书籍结构优化）

**2.2 上传书籍文档**

支持格式：PDF、TXT、DOCX、EPUB

上传建议：
- 每本书单独上传，便于引用时显示具体书名
- 命名规范：`作者姓名_书名_年份.pdf`（示例：`Spurgeon_Morning-Evening_1865.pdf`）
- PDF 质量：有文字层的 PDF 解析效果远优于纯扫描版

**2.3 调整分块策略**

RAGFlow 默认分块大小适合通用文档，神学书籍建议调整：

| 参数 | 默认值 | 建议值 | 原因 |
|------|-------|-------|------|
| Chunk size | 512 tokens | 768 tokens | 神学论述段落较长，切太短丢失上下文 |
| Overlap | 50 tokens | 100 tokens | 确保段落衔接处不丢失关键信息 |
| Parse method | General | Book | 识别章节结构，保留层级关系 |

**2.4 配置检索参数**

```
Top-K（返回片段数）：5~8
相似度阈值：0.3（神学术语检索不宜设太高）
Reranking：开启（提高引用精准度）
```

---

### Phase 3：配置 GraphRAG（第 3~5 天）

GraphRAG 负责构建书籍间的概念关联图。

```bash
# 安装 Microsoft GraphRAG
pip install graphrag

# 初始化项目
mkdir christian-graphrag && cd christian-graphrag
python -m graphrag init

# 配置（编辑 settings.yml）
# 指定 LLM 为 DeepSeek V3
# 指定输入目录为书籍文本文件夹
```

**配置 `settings.yml` 关键项**：

```yaml
llm:
  api_key: ${DEEPSEEK_API_KEY}
  type: openai_chat
  model: deepseek-chat
  api_base: https://api.deepseek.com

input:
  type: file
  file_type: text
  base_dir: ./books/      # 存放书籍 TXT 文件的目录

entity_extraction:
  entity_types:
    - person              # 神学家、圣经人物
    - concept             # 教义概念（恩典、救赎、预定论...）
    - scripture           # 圣经章节引用
    - work                # 书名、文献
```

```bash
# 建立索引（耗时较长，取决于书籍数量）
python -m graphrag index --root ./

# 完成后可执行图谱查询
python -m graphrag query --root ./ \
  --method global \
  --query "加尔文和阿民念主义对预定论的不同立场"
```

### Phase 4：整合与测试（第 5~7 天）

**4.1 创建 RAGFlow 对话助手**

1. `Chat → Create Assistant`
2. 关联知识库：`Christian Literature`
3. 配置 System Prompt：

```
你是一个基督徒书报研究助手。你只能基于知识库中提供的书目内容回答问题。

规则：
1. 每条回答必须引用具体来源（书名 + 章节/页码）
2. 如果知识库中没有相关内容，明确回答："在现有书目中未找到关于此问题的内容"
3. 不使用你自身训练数据中的神学知识来补充回答
4. 如果不同书目对同一问题有不同观点，客观呈现各方立场，不主动做出神学判断
5. 使用提问者的语言（中文问则中文答，英文问则英文答）
```

**4.2 测试用例**

| 测试类型 | 示例问题 | 期望行为 |
|---------|---------|---------|
| 基础检索 | "约翰福音第三章的注释是什么？" | 返回相关书目的注释内容 + 来源 |
| 封闭性测试 | "量子力学的基本原理是什么？" | 明确回答"知识库中无此内容" |
| 跨书对比 | "不同作者如何解释'因信称义'？" | 列出多本书的观点并标注来源 |
| 语言切换 | 用英文提问中文书籍内容 | 用英文回答，引用中文书籍 |
| 幻觉测试 | 询问知识库中确实没有的神学家 | 不编造内容，明确说明未找到 |

### Phase 5：上线与维护

**添加新书籍**：直接在 RAGFlow 知识库中上传，系统自动解析和索引，无需重新配置。

**GraphRAG 更新**：添加书籍后需重新运行 `graphrag index` 更新图谱（建议批量添加后统一更新）。

**监控指标**：
- 引用命中率（回答是否真的来自书目）
- 用户对回答质量的反馈
- 幻觉频率（回答了知识库中没有的内容）

## 8. 预期结果与局限性

### 8.1 预期能达到的效果

| 功能 | 预期表现 |
|------|---------|
| 单书检索 | 准确找到相关段落，引用精准，95% 以上的问题能给出有来源的回答 |
| 跨书对比 | 能梳理同一主题在不同书目中的观点，适合教义研究 |
| 封闭性 | 知识库外的问题明确拒答，不编造内容 |
| 语言切换 | 中英文提问均能自然回答，引用混合语言书目无障碍 |
| 响应速度 | API 方案约 3~8 秒/次，本地方案约 5~15 秒/次 |

### 8.2 已知局限性

**文档质量依赖**：扫描版 PDF 若 OCR 质量差，解析后的文本会有错误，直接影响检索准确性。建议优先使用有文字层的 PDF 或 EPUB 格式。

**长文推理上限**：当问题涉及跨越多章的神学论证时，RAGFlow 检索到的片段可能无法提供完整论证链，GraphRAG 可以部分弥补，但无法完全解决。

**非文字内容**：书中的图表、圣经原文附图等无法被检索（仅限文字内容）。

**语义边界**：同一概念在不同书目中可能用不同术语表达（如"预定论" vs "拣选论"），需要在初期测试中识别并配置同义词。

### 8.3 未来扩展路径

```
现阶段（中文 + 英文）
    ↓
添加西班牙语、韩语书籍
  → 无需换模型（DeepSeek V3 / Qwen3 原生支持）
  → 更新 Embedding 模型为多语言版本（bge-m3 已支持）
    ↓
可选：微调 Embedding 模型
  → 用领域词汇（神学术语）优化检索精度
  → 需要 UltraRAG 或 LlamaIndex 的训练模块
```
