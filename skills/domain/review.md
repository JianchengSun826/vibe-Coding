# review（Domain Skill）

> 系统性地审查代码质量、正确性、安全性和可维护性。

**类型**：领域类 — 按需查阅
**触发时机**：审查 PR、结对编程、代码质量检查

---

## 审查顺序（优先级从高到低）

```
1. 正确性    ← 有没有 bug？逻辑对吗？
2. 安全性    ← 有没有漏洞？（配合 /security skill）
3. 测试      ← 测试充分吗？边界情况覆盖了吗？
4. 可维护性  ← 别人能看懂和修改吗？
5. 性能      ← 有没有明显的性能问题？（配合 /perf skill）
```

**先看正确性，再看风格。** 不要为了格式问题争论，先确保功能是对的。

---

## 1. 正确性审查

检查清单：
- [ ] 核心算法/逻辑有没有 bug？
- [ ] 边界条件处理了吗（空值、0、最大值、空列表）？
- [ ] 错误处理完整吗（网络失败、数据库超时、无效输入）？
- [ ] 并发场景下有没有竞争条件？
- [ ] 资源有没有正确释放（数据库连接、文件句柄）？

**示例评论**：

```typescript
// 代码
function divide(a: number, b: number): number {
  return a / b;
}

// Review 评论：
// "这里没有处理 b=0 的情况，会返回 Infinity。
//  建议加：if (b === 0) throw new Error('Division by zero')"
```

```typescript
// 代码
const items = await cache.get('items') ?? await db.getItems();
// 写入缓存的代码在别处

// Review 评论：
// "如果 db.getItems() 查询出来的结果没有写入缓存，
//  每次 cache miss 都会查数据库，缓存形同虚设。
//  建议在这里加：if (!cached) cache.set('items', items, TTL);"
```

---

## 2. 安全性审查

（详见 `/security` skill，这里只列关键检查项）

- [ ] 用户输入有没有验证？
- [ ] SQL 查询有没有参数化？
- [ ] 权限检查是否在每个需要保护的端点都有？
- [ ] 有没有硬编码的密钥？
- [ ] 错误信息有没有泄露敏感信息？

---

## 3. 测试审查

- [ ] 正常路径有测试？
- [ ] 失败路径有测试？（比登录成功更重要的是登录失败）
- [ ] 测试覆盖了边界条件？
- [ ] 测试是否真的在测业务逻辑（而不只是在测 mock）？

**示例评论**：

```typescript
// 代码中只有一个测试
test('should create user', async () => {
  const user = await createUser('test@example.com', 'password');
  expect(user.id).toBeDefined();
});

// Review 评论：
// "当前只测试了成功场景。建议补充：
//  - 重复邮箱应该返回什么错误？
//  - 密码太短（< 8 位）应该怎么处理？
//  - email 格式无效应该怎么处理？
//  这些失败场景的测试通常比成功场景更重要。"
```

---

## 4. 可维护性审查

- [ ] 函数名和变量名能清楚表达意图？
- [ ] 函数是否只做一件事？
- [ ] 复杂逻辑有注释吗？
- [ ] 有没有重复代码可以提取？
- [ ] 魔法数字有没有命名常量？

**示例评论**：

```typescript
// 代码
if (user.role === 2) {  // 2 是什么？
  ...
}

// Review 评论：
// "这里的 2 是什么含义？建议用常量：
//  const ROLE_ADMIN = 2; 或者用枚举。"
```

```typescript
// 代码
function process(data: any) {
  // 200 行代码：验证 + 转换 + 存储 + 发通知
}

// Review 评论：
// "这个函数做了太多事情，建议拆分成：
//  validateData()、transformData()、saveToDatabase()、sendNotification()
//  每个函数单独测试也更容易。"
```

---

## 5. 性能审查

- [ ] 有没有 N+1 查询？
- [ ] 有没有在循环里做昂贵操作？
- [ ] 大数据集有没有分页？
- [ ] 是否在每次请求都重复计算可以缓存的结果？

---

## 如何给好的 Review 评论

| 类型 | 格式 |
|------|------|
| Bug | "这里有个 bug：[描述问题]。[建议修复方式]" |
| 建议（非强制） | "建议（nit）：可以考虑 [建议]，这样 [好处]。" |
| 疑问 | "我不太理解这里 [xxx] 的设计，能解释一下原因吗？" |
| 表扬 | "这个处理方式很聪明！" |

**原则**：
- 评论代码，不评论人（"这行代码有问题" 而不是 "你写错了"）
- 解释为什么是问题，而不只是说"这样不对"
- 区分必须改（bug）和建议改（风格/优化）
