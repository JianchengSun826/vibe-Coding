# brainstorming

> 在动手实现之前，先探索需求方向、评估方案、对齐期望。

**类型**：流程类（Superpowers）— 不可跳过
**触发时机**：接到任何新功能需求、修改请求、架构决策

---

## 为什么需要它

没有 brainstorming 的后果：直接开始实现，做了一半才发现方向错了。

> 用户说："帮我加个搜索功能"
>
> 没有 brainstorming → AI 实现了前端模糊匹配
> 用户说："我要的是全文搜索，能搜数据库里的内容"
> → 重做

有了 brainstorming → 先问清楚：是前端筛选还是后端搜索？需要模糊匹配吗？数据量多大？→ 一次做对。

---

## 这个 skill 会做什么

1. **理解用户意图**：不只是字面需求，还有背后的目标
2. **探索现有代码**：了解当前架构，找到最合适的切入点
3. **提出多个方案**：列出 2~3 种实现方式，分析各自的权衡
4. **对齐方向**：让用户选择方案，再开始实现

---

## 示例

### 示例 1：功能开发

```
用户："帮我实现用户登录功能"

brainstorming 会问：
- 登录方式：邮箱密码 / Google OAuth / 手机号验证码？
- Token 方案：JWT（无状态）还是 Session（有状态）？
- 有效期：Token 多久过期？需要 refresh token 吗？
- 已有用户表了吗？还是需要新建？
- 是否需要"记住我"功能？

用户回答后，AI 会说：
"根据你的需求，我建议用 JWT + bcrypt 方案，原因是..."
然后进入 writing-plans。
```

### 示例 2：技术决策

```
用户："我们的 API 越来越慢，帮我优化一下"

brainstorming 会问：
- 哪个接口慢？慢多少？有监控数据吗？
- 是所有请求慢，还是特定操作慢？
- 用的是什么数据库？有没有做过索引优化？
- 最近有没有数据量暴增？

这些问题防止 AI 盲目"优化"一个不是瓶颈的地方。
```

### 示例 3：架构决策

```
用户："帮我把单体应用拆成微服务"

brainstorming 会问：
- 当前遇到了什么具体问题（部署慢？扩展难？团队协作？）？
- 团队规模多大？微服务的运维成本是否可以承担？
- 是否考虑过中间方案（模块化单体）？

可能的结论：
"根据你的团队规模（3人）和当前痛点（部署慢），
建议先做模块化拆分，而不是直接上微服务。原因是..."
```

## 常见误区

| 错误想法 | 正确做法 |
|---------|---------|
| "需求很清楚了，直接做就行" | 「清楚」的需求经常有未言明的假设，brainstorming 会发现它们 |
| "问太多问题会烦用户" | 一次问对，比做错重做要好得多 |
| "先做出来再说，不合适再改" | 改比做更贵，尤其涉及数据库设计时 |
