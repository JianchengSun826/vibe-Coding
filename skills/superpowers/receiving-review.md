# receiving-code-review

> 收到他人的代码审查意见时，理性分析，既不盲目接受，也不防御性拒绝。

**类型**：流程类（Superpowers）— 不可跳过
**触发时机**：收到 PR review 评论、结对编程反馈、代码审查意见

---

## 处理流程

```
1. 理解意见   ← 确认自己真的看懂了 reviewer 的意图
2. 验证技术   ← 评估建议是否技术上正确或合理
3. 决策       ← 接受 / 讨论 / 解释
4. 回复       ← 无论哪种决策，都给出明确的回应
```

---

## 三种处理方式

### 方式 1：理解后接受（技术正确）

```
Reviewer：
  "这里有 N+1 查询问题，在循环里调用数据库会很慢。
   建议用 JOIN 一次性拿到所有数据。"

处理步骤：
  1. 理解：N+1 是指每次循环都发一个数据库请求
  2. 验证：查看代码，确认确实是循环里调用数据库
  3. 查 /db skill 确认 JOIN 的正确写法
  4. 接受并修改

回复：
  "你说得对，这里确实是 N+1 问题，我改用 JOIN 了。感谢指出！"
```

---

### 方式 2：理解后讨论（有不同看法）

```
Reviewer：
  "这个函数太长了（80行），应该拆成 5 个更小的函数。"

处理步骤：
  1. 理解 reviewer 的关切：函数过长，可维护性差
  2. 评估：这 80 行是否有内聚性？
     - 如果这 5 个步骤必须按顺序执行，拆开后调用者需要了解内部顺序
     - 实际上可能增加认知负担
  3. 有不同意见，但可以找中间方案

回复：
  "我理解你对函数长度的顾虑。这 5 个步骤有严格的顺序依赖，
  拆分后调用者需要了解执行顺序，反而可能更难理解。
  我可以在函数内加注释来分隔各个阶段，提高可读性。
  如果你觉得长度仍然是问题，我们可以把它拆成 2 个（而不是 5 个）
  更有意义的阶段。你觉得哪种方式更好？"
```

---

### 方式 3：不盲目接受（reviewer 可能有误）

```
Reviewer：
  "不需要这么复杂的类型定义，用 any 就好了。"

处理步骤：
  1. 理解意图：可能是觉得类型定义太繁琐
  2. 技术评估：
     - any 会丢失类型安全
     - TypeScript 编译器无法在这里捕获错误
     - 这个函数是核心业务逻辑，类型安全很重要
  3. 不接受，但用事实解释原因

回复：
  "我理解 any 写起来更简单，但这里的类型定义有具体作用：
  它让编译器在这里的调用处帮我们捕获类型错误。
  上次这里就因为传错类型导致了 bug，所以我特意加了类型约束。
  你是否有具体的场景觉得这个类型太繁琐？如果有的话我们可以讨论简化方案。"
```

---

## 坏的回复模式

| 类型 | 示例 | 问题 |
|------|------|------|
| 盲目接受 | "好的好的，我马上改" | 没有理解就执行，可能引入新问题 |
| 防御性拒绝 | "你不懂我的代码" | 关闭讨论，错过学习机会 |
| 敷衍 | "嗯，我看看" | 不给明确回复，让 reviewer 不知道意见有没有被处理 |
| 过度道歉 | "对不起这里写得很烂..." | 自我否定，而不是讨论技术 |

---

## 好的回复模式

```
"我理解你的意思，[复述 reviewer 的关切]。
[你的评估/不同意见]。
[你的方案/建议]。"
```

回复要包含：
1. 表示已理解对方的意图（不是敷衍，是真的理解）
2. 说明你的技术判断
3. 提出下一步行动
